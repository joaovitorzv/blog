Git Workflow em time

23 de novembro de 2021

Commitar direto na main não é um problema quando você está desenvolvendo sozinho, porém, em um contexto onde existem outras pessoas trabalhando no mesmo projeto é preciso seguir outro caminho, esse caminho varia de acordo com o workflow que o time definiu mas não é algo tão distante do seguinte workflow

  1. Criar uma Branch (para cada task)
  2. Commit e Push
  3. Pull Request
  4. Solicitar Review
  5. Merge

Vamos imaginar que você pegou uma task que adiciona um drop-down menu, como era o seu primeiro dia e você estava ansioso, após escrever algumas linhas de código percebeu que não tinha criado a branch para fazer essa task e ainda estava na main, nesse caso como ainda não a criamos podemos cria-la e mudar

Existem outros edge-cases para este caso (caso já tenha commitado as alterações), se você quiser ir mais a fundo neste link tem uma explicação completa.

para criar a nova branch:

git branch new-drop-down

e para trocar de branch:

git checkout new-drop-down

// a palavra checkout pode confundir as vezes 
// o comando acima também poderia ser assim:
git switch new-drop-down

e se você quiser criar a branch e trocar de uma vez só:

git checkout -b new-drop-down

Pronto agora você já finalizou a task e precisa commitar, uma boa mensagem de commit precisa ser objetiva para que outras pessoas possam entender do que se trata aquele commit. Para deixar a mensagem bem legível vamos formata-lo da seguinte maneira (pessoalmente eu mantenho tudo em inglês):

<tipo>[escopo opcional]: <descrição>
  • tipos mais utilizados:
    • fix: corrigir um bug
    • feat: introduzir uma feature
    • chore: alterações em configurações, ferramentas e alterações em coisas que não vão para produção (por exemplo uma alteração no .gitignore)
    • docs: alterações na documentação
    • style: alterações nos estilos (css)
    • refactor: código refatorado
    • perf: alreações para melhoria de desempenho
    • test: esse é bem auto-explicativo, testes

com isso em mente podemos commitar a task que acabamos de fazer

git add .
⁠git commit -m "feat(header): new drop-down menu"

// ou se preferir fazer tudo de uma vez
// usando a opção -a significa -all a qual diz para o git 
// colocar em stage em todos arquivos modificados/deletados 

⁠git commit -a -m "feat(header): new drop-down menu"

agora precisamos fazer o Pull Request, pessoalmente eu acho o nome que o Gitlab utiliza para se referir a 'Pull Request' bem mais intiutivo: Merge Request

podemos fazer da seguinte maneira

  1. Clicar no link que o git nos deu aqui em cima👆
  2. Ou no github ao abrir o respositório que foi feito o push terá um botão Compare & Pull Request ⁠

ao clicar no botão teremos que adicionar um Titulo e Descrição descrevendo o que fizemos

na imagem acima logo ao lado direito mostra que precisamos de pelo menos 1 review aprovado para fazer o merge, basta clicar ali e selecionar alguém para fazer o review

Se você precisar fazer alterações no código basta commitar na mesma branch e todos os commits serão direcionados para esse PR.

após o review ser aprovado você poderá fazer o 'squash & merge'

Utilizar o squash é como se pegassemos todos os commits que fizemos nesta branch e juntassemos em um só, assim o histórico de PR's fica mais limpo e organizado.

depois de ter o merge feito vai aparecer uma opção para deletar essa branch, podemos deletar pois o historico de todas alterações/commits ficarão salvos nos closed PR's

voltando para o ambiente de desenvolvimento podemos trocar de branch para a main

git checkout main

⁠Switched to branch 'main'
⁠Your branch is up to date with 'origin/main'.

e puxar todas as alterações que acabamos de fazer

git pull
